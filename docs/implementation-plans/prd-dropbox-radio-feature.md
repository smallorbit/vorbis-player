# PRD: Radio Feature (Dropbox Provider)

**Version:** 0.1 — Initial Dropbox-only implementation
**Status:** Draft
**Date:** 2026-03-08

---

## 1. Overview

### 1.1 Problem

Vorbis Player users with a Dropbox music library have no way to discover and play music similar to what they're currently listening to. Spotify's native Radio feature exists in the Spotify client but is not exposed through its API, and Spotify's Recommendations endpoint has been deprecated (Nov 2024). Users who primarily use the Dropbox provider have no recommendation or radio functionality at all.

### 1.2 Solution

Add a "Radio" feature to the Dropbox provider that generates a queue of similar tracks from the user's own Dropbox library, seeded from a track, artist, or album. The feature uses the Last.fm API to find similar music and matches results against the locally indexed Dropbox catalog, using MusicBrainz IDs (MBIDs) and normalized name matching.

### 1.3 Scope

This PRD covers the Dropbox provider only. Spotify provider radio will be addressed in a future iteration and is explicitly out of scope.

### 1.4 Success Criteria

- User can start a radio session from any track, artist, or album in their Dropbox library.
- The generated queue contains tracks from the user's Dropbox library that are meaningfully related to the seed.
- Radio queue is populated within 2 seconds of initiation for a typical library (1,000–10,000 tracks).
- At least 60% of radio seeds produce a non-empty queue (for a well-tagged library of 2,000+ tracks with diverse artists).

## 2. User Experience

### 2.1 Entry Points

There are three ways a user initiates a radio session:

**From a track:** User is listening to a track (or viewing one in a playlist). They trigger "Start Radio" and get a queue of similar tracks from their library.

**From an artist:** User selects "Start Radio" from an artist context (e.g., clicking an artist name). The queue contains tracks similar to that artist's top tracks, filtered to what exists in their library.

**From an album:** User selects "Start Radio" from an album context. The queue is generated by aggregating similar tracks for all tracks on the album, filtered to what exists in their library.

### 2.2 UI Trigger

The exact UI placement is flexible, but the recommended approach is:

- **Context menu or long-press option** on tracks, artist names, and albums that reads "Start Radio" or shows a radio/antenna icon.
- **Alternatively:** A radio icon button added to the player controls area when Dropbox is the active provider.

The trigger should be unobtrusive and consistent with the existing flip-menu / bottom-bar interaction patterns.

### 2.3 Radio Session Behavior

Once initiated:

1. The current playback queue is replaced with the radio-generated queue.
2. The first track in the radio queue begins playing immediately.
3. Shuffle is enabled by default for radio sessions (the queue is already curated, but shuffle adds variety to ordering).
4. The playlist drawer shows the radio queue with a visual indicator that this is a "Radio" session (e.g., a label or icon).
5. The user can skip tracks, like/unlike tracks, and use all normal playback controls.
6. The radio session ends when the queue is exhausted or the user navigates back to the library and selects a different album/playlist.

### 2.4 Edge Cases

**Empty results:** If no matching tracks are found in the Dropbox library, show a brief toast/notification: "No similar tracks found in your library." Do not change the current playback.

**Partial results:** If fewer than 5 tracks are matched, still start the radio session but inform the user: "Found N similar tracks in your library."

**Seed track not on Last.fm:** If Last.fm returns no results for the seed (obscure or mistagged track), fall back to artist radio — find similar artists and pull their tracks from the library. If that also fails, show the empty-results message.

## 3. Technical Architecture

### 3.1 System Overview

```
┌─────────────────────────────────────────────────────┐
│  UI Layer                                           │
│  (Radio button / context menu)                      │
└──────────────────┬──────────────────────────────────┘

┌─────────────────────────────────────────────────────┐

│  - Orchestrates Last.fm calls                       │

└──────┬───────────────────┬──────────────────────────┘
       │                   │
       ▼                   ▼
┌──────────────┐   ┌──────────────────────────────────┐
│  LastFmClient│   │  CatalogMatcher                  │
│  (API calls) │   │  - Matches Last.fm results       │
│              │   │    against Dropbox catalog        │
│              │   │  - Uses MBID index (primary)      │
│              │   │  - Falls back to name matching    │
└──────────────┘   └──────────────────────────────────┘
                           │
                           ▼
```

### 3.2 New Modules

#### 3.2.1 `LastFmClient` — `src/services/lastfm.ts`

A thin, stateless API client for Last.fm. No user authentication required.

**Responsibilities:**
- Make HTTP requests to the Last.fm API.
- Parse JSON responses.
- Handle errors and rate limiting (5 req/sec ceiling).

**Configuration:**
- API key stored in `VITE_LASTFM_API_KEY` environment variable.
- User-Agent header: `VorbisPlayer/1.0`.

**Methods:**

```typescript
interface LastFmSimilarTrack {
  name: string;
  artist: string;
  artistMbid: string | null;
  trackMbid: string | null;
  matchScore: number; // 0.0–1.0
}

interface LastFmSimilarArtist {
  name: string;
  mbid: string | null;
  matchScore: number;
}

interface LastFmClient {
  getSimilarTracks(artist: string, track: string, limit?: number): Promise<LastFmSimilarTrack[]>;
  getSimilarArtists(artist: string, limit?: number): Promise<LastFmSimilarArtist[]>;
  getArtistTopTracks(artist: string, limit?: number): Promise<LastFmSimilarTrack[]>;
  getAlbumTracks(artist: string, album: string): Promise<{ name: string; artist: string }[]>;
}
```

**Notes:**
- All methods accept artist/track names as strings. MBID-based lookups can be added later as an optimization.
- `limit` defaults to 50 for `getSimilarTracks`, 20 for `getSimilarArtists`, 10 for `getArtistTopTracks`.
- Responses are normalized into the interfaces above; raw Last.fm response shapes are not leaked.

#### 3.2.2 `CatalogMatcher` — `src/services/catalogMatcher.ts`

Matches external track references (from Last.fm) against the local Dropbox catalog.

**Responsibilities:**
- Build and maintain in-memory indexes over the Dropbox catalog for fast lookups.
- Match by MBID (primary, exact match).
- Match by normalized artist + track name (fallback, fuzzy).
- Deduplicate results and exclude the seed track.

**Indexes (built once when Dropbox catalog is loaded or updated):**

```typescript
interface CatalogIndexes {
  // Primary: MBID-based lookup
  byRecordingMbid: Map<string, DropboxTrack>;     // MUSICBRAINZ_TRACKID → track
  byArtistMbid: Map<string, DropboxTrack[]>;       // MUSICBRAINZ_ARTISTID → tracks

  // Fallback: name-based lookup
  byNormalizedKey: Map<string, DropboxTrack[]>;    // normalizedArtist + normalizedTitle → tracks
  byNormalizedArtist: Map<string, DropboxTrack[]>; // normalizedArtist → tracks
}
```

**Name normalization function:**

```typescript
function normalizeForMatching(input: string): string {
  return input
    .toLowerCase()
    .normalize('NFD')
    .replace(/[\u0300-\u036f]/g, '')       // strip accents
    .replace(/\s*\(.*?\)\s*/g, '')         // strip parentheticals: (Remastered), (feat. X)
    .replace(/\s*\[.*?\]\s*/g, '')         // strip brackets: [Deluxe Edition]
    .replace(/\s*-\s*(single|ep|remix)$/i, '') // strip trailing suffixes
    .replace(/[^a-z0-9]/g, '')             // strip non-alphanumeric
    .trim();
}
```

**Match method:**

```typescript
interface MatchResult {
  track: DropboxTrack;
  confidence: 'mbid' | 'name-exact' | 'name-fuzzy';
  lastfmMatchScore: number;
}

function matchTrack(
  candidate: LastFmSimilarTrack,
  indexes: CatalogIndexes
): MatchResult | null;
```

Match priority:
1. If candidate has `trackMbid`, look up in `byRecordingMbid`. If found → confidence `'mbid'`.
2. Compute `normalizeForMatching(candidate.artist) + normalizeForMatching(candidate.name)`. Look up in `byNormalizedKey`. If found → confidence `'name-exact'`.
3. (Future/optional) Levenshtein distance on normalized key with threshold ≤ 2 → confidence `'name-fuzzy'`.

#### 3.2.3 `RadioService` — `src/services/radioService.ts`

Orchestrator that ties LastFmClient and CatalogMatcher together.

**Responsibilities:**
- Accept a radio seed and produce an ordered queue of Dropbox tracks.
- Handle the three seed types (track, artist, album) with different strategies.
- Deduplicate, sort by relevance, and cap the queue.

**Interface:**

```typescript
type RadioSeed =
  | { type: 'track'; artist: string; track: string; mbid?: string }
  | { type: 'artist'; artist: string; mbid?: string }
  | { type: 'album'; artist: string; album: string; tracks: { artist: string; name: string }[] };

interface RadioResult {
  queue: DropboxTrack[];
  seedDescription: string;  // e.g., "Radio based on Creep by Radiohead"
  matchStats: {
    lastfmCandidates: number;
    matched: number;
    byMbid: number;
    byName: number;
  };
}

interface RadioService {
  generateQueue(seed: RadioSeed): Promise<RadioResult>;
}
```

**Strategy per seed type:**

**Track seed:**
1. Call `lastfm.getSimilarTracks(artist, track, 100)`.
2. Match each result against catalog.
3. If fewer than 5 matches, also call `lastfm.getSimilarArtists(artist, 10)`, then for each similar artist found in the catalog, add their tracks (up to 3 per artist).
4. Deduplicate, sort by `lastfmMatchScore` descending, cap at 50 tracks.

**Artist seed:**
1. Call `lastfm.getSimilarArtists(artist, 30)`.
2. For each similar artist, check `byArtistMbid` or `byNormalizedArtist` in catalog.
3. For matched artists, include up to 5 of their tracks (randomly selected or by track popularity if available).
4. Also include tracks from the seed artist themselves (mixed in).
5. Deduplicate, cap at 50 tracks.

**Album seed:**
1. Take the album's track list (passed in as `tracks`).
2. For each track (up to 5 sampled from the album), call `lastfm.getSimilarTracks`.
3. Aggregate all results, deduplicate, match against catalog.
4. Exclude tracks from the seed album itself.
5. Sort by aggregate match score, cap at 50 tracks.

### 3.3 Catalog Index Integration

The Dropbox provider already scans the full library at startup and caches metadata in IndexedDB. The catalog index for radio matching should be built **at the same time**, as a post-processing step after the catalog is loaded.

**Where it fits:**
- After the Dropbox catalog adapter finishes loading/scanning albums and tracks, call `CatalogMatcher.buildIndexes(allTracks)`.
- The indexes live in memory (not persisted to IndexedDB) since they're cheap to rebuild from the already-cached catalog.
- If the catalog is updated (e.g., background sync adds new tracks), rebuild the indexes.

**Tag extraction:**
The existing Dropbox provider reads ID3 tags for title, artist, album, and cover art. The following additional tags need to be extracted (if present) and stored on the track object:

| ID3v2 Frame | Vorbis Comment | Field Name |
|---|---|---|
| `TXXX:MusicBrainz Release Track Id` or `UFID` | `MUSICBRAINZ_TRACKID` | `musicbrainzRecordingId` |
| `TXXX:MusicBrainz Artist Id` | `MUSICBRAINZ_ARTISTID` | `musicbrainzArtistId` |
| `TSRC` | `ISRC` | `isrc` |

These should be optional fields on the track type. Most Picard-tagged files will have them; untagged files will not.

### 3.4 Data Flow: End to End

```
User taps "Start Radio" on track "Creep" by "Radiohead"
│
├─ RadioService.generateQueue({ type: 'track', artist: 'Radiohead', track: 'Creep' })
│
├─ LastFmClient.getSimilarTracks('Radiohead', 'Creep', 100)
│   └─ HTTP GET ws.audioscrobbler.com/2.0/?method=track.getsimilar&...
│   └─ Returns: [{ name: 'Fake Plastic Trees', artist: 'Radiohead', matchScore: 0.91 },
│                { name: 'Paranoid Android', artist: 'Radiohead', matchScore: 0.87 },
│                { name: 'Karma Police', artist: 'Radiohead', matchScore: 0.82 },
│                { name: 'No Surprises', artist: 'Radiohead', matchScore: 0.78 },
│                { name: 'Lucky', artist: 'Morcheeba', matchScore: 0.45 },
│                ... 95 more ]
│
├─ CatalogMatcher.matchTrack() for each candidate
│   ├─ 'Fake Plastic Trees' → byRecordingMbid hit → ✅ matched (confidence: mbid)
│   ├─ 'Paranoid Android'   → byNormalizedKey hit  → ✅ matched (confidence: name-exact)
│   ├─ 'Lucky' by Morcheeba → no MBID, no name hit → ❌ not in library
│   └─ ... continue for all candidates
│
├─ Dedup, sort by matchScore, cap at 50
│
└─ Return RadioResult {
     queue: [DropboxTrack, DropboxTrack, ...],
     seedDescription: 'Radio based on Creep by Radiohead',
     matchStats: { lastfmCandidates: 100, matched: 23, byMbid: 14, byName: 9 }
   }
```

## 4. Environment & Configuration

### 4.1 New Environment Variable

```env
VITE_LASTFM_API_KEY="your_lastfm_api_key_here"
```

- Obtained free from https://www.last.fm/api/account/create
- Added to `.env.local` and `.env.example`
- If not set, the radio feature is hidden/disabled (same pattern as the Dropbox provider when `VITE_DROPBOX_CLIENT_ID` is absent).

### 4.2 No New Backend Required

All API calls (Last.fm) are made directly from the browser. Last.fm's API supports CORS and requires only an API key (no secret). No proxy or serverless function is needed.

## 5. New Dependencies

| Package | Purpose | Size |
|---|---|---|
| None required | Last.fm API is simple enough to call with `fetch` directly | — |

The Last.fm API is a single REST endpoint with query parameters. A dedicated npm package (like `last-fm`) is optional. For this initial implementation, a lightweight custom client using `fetch` is recommended to keep the dependency footprint at zero.

## 7. Testing Strategy

### 7.1 Unit Tests

**`lastfm.test.ts`:**
- Mock `fetch` responses with realistic Last.fm JSON payloads.
- Test parsing of `track.getSimilar`, `artist.getSimilar`, `artist.getTopTracks` responses.
- Test error handling: 400 errors, rate limiting, empty results, malformed responses.

**`catalogMatcher.test.ts`:**
- Build indexes from a fixture set of mock Dropbox tracks (some with MBIDs, some without).
- Test MBID matching: exact match, no match.
- Test name matching: exact normalized match, partial match, no match.
- Test normalization edge cases: accented characters, parentheticals, case differences, feat. credits.
- Test deduplication: same track from different albums.

**`radioService.test.ts`:**
- Mock LastFmClient and CatalogMatcher.
- Test track seed strategy: happy path, fallback to artist radio when few matches.
- Test artist seed strategy: similar artists with catalog intersection.
- Test album seed strategy: aggregation across album tracks.
- Test edge cases: empty Last.fm results, zero catalog matches, seed track excluded from results.

### 7.2 Integration Tests

- End-to-end test with a small fixture library (10–20 tracks) and real Last.fm API calls (using a test API key).
- Verify that a radio seed for a well-known artist (e.g., "Radiohead") produces non-empty results when the fixture library contains known similar artists.

### 7.3 Manual Testing Checklist

- [ ] Start radio from a track → queue populates, first track plays
- [ ] Start radio from an artist → queue populates with tracks from similar artists
- [ ] Start radio from an album → queue populates with tracks similar to the album's tracks
- [ ] Radio with obscure/untagged track → graceful fallback or empty-results message
- [ ] Radio queue exhausted → playback stops, user can navigate back to library
- [ ] Start new radio while radio is already playing → replaces the queue
- [ ] `VITE_LASTFM_API_KEY` not set → radio button/option is hidden

## 8. Implementation Plan

### Phase 1: Foundation (Services Layer)

**Goal:** Build and test the three new service modules with no UI changes.

1. **Define types** — `src/types/radio.ts` with `RadioSeed`, `RadioResult`, `LastFmSimilarTrack`, etc.
2. **Implement `LastFmClient`** — Fetch wrapper with response parsing and error handling.
3. **Implement `CatalogMatcher`** — Index builder and matching logic.
4. **Implement `RadioService`** — Orchestration for all three seed types.
5. **Write unit tests** for all three modules.
6. **Extend Dropbox track type** — Add optional `musicbrainzRecordingId`, `musicbrainzArtistId`, and `isrc` fields. Update the Dropbox catalog adapter's ID3 tag extraction to populate them.

**Validation:** Run unit tests. Manually call `radioService.generateQueue()` from the browser console against a real Dropbox library to verify end-to-end behavior.

### Phase 2: Hook & Minimal UI

**Goal:** Make the feature usable through the UI.

1. **Create `useRadio` hook** — Manages radio state (active/inactive, current seed, queue). Integrates with the existing playback hooks to set the queue and start playback.
2. **Add radio trigger** — A "Start Radio" option accessible from the player (e.g., in the flip menu, bottom bar, or as a context action on track/artist/album names).
3. **Radio queue indicator** — Visual indicator in the playlist drawer that the current queue is a radio session (e.g., an icon + label like "Radio: Creep by Radiohead").
4. **Disable when unconfigured** — Hide radio UI when `VITE_LASTFM_API_KEY` is not set.

### Phase 3: Polish & Edge Cases

**Goal:** Handle edge cases and improve the experience.

1. **Loading state** — Show a brief loading indicator while the radio queue is being generated.
2. **Empty/partial results messaging** — Toast notifications for empty or small queues.
3. **Queue extension** — When the queue has fewer than 5 tracks remaining, automatically fetch more similar tracks (infinite radio).
4. **Match stats (dev only)** — Optionally log `matchStats` to console for debugging catalog coverage.

## 9. Future Considerations (Out of Scope)

These items are explicitly deferred:

- **Spotify provider radio** — Same RadioService but matching against Spotify Search API instead of the local catalog.
- **MusicBrainz ISRC enrichment** — Pre-fetching ISRCs from MusicBrainz for the entire catalog to improve matching (constrained by 1 req/sec rate limit; requires background batch job).
- **Hybrid radio** — Mixing Dropbox and Spotify results in a single radio session.
- **User feedback loop** — Using likes/skips to refine radio recommendations over time.
- **Offline radio** — Pre-computing radio queues for offline use.
- **Audio-based similarity** — Using AcoustID or audio fingerprinting for similarity instead of/in addition to Last.fm co-listen data.

## 10. Open Questions

1. **UI placement:** Where does the "Start Radio" trigger live? Options include the album art flip menu, a new bottom bar button, context menus on artist/album names, or a combination. This should be decided based on what feels most natural with the existing interaction patterns.

2. **Queue replacement vs. append:** When starting radio, should it always replace the current queue, or should there be an option to append radio tracks after the current queue? The initial implementation replaces the queue for simplicity.

3. **Seed track inclusion:** Should the seed track itself be included in the radio queue (e.g., as the first track)? Or should radio always play only *other* tracks? Recommendation: include the seed track as the first item so playback starts immediately with something the user already likes, then transition into similar tracks.

4. **Max concurrent Last.fm calls:** For album-based radio, should we call `getSimilarTracks` for all album tracks in parallel (fast, but could hit rate limits on large albums) or sequentially (slow, but safe)? Recommendation: parallel with a concurrency limit of 3.
